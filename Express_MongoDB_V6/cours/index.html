⚡ Présentation d'Express.js
Découvrez ce qu'est Express.js, pourquoi l'utiliser et comment il simplifie le développement web avec Node.js.

🤔 Qu'est-ce qu'Express.js ?

Express.js est un framework web minimaliste et flexible pour Node.js. Il fournit un ensemble de fonctionnalités robustes pour créer des applications web et des API.

En clair : Node.js vous donne le moteur, Express vous donne la voiture complète avec volant, freins et tableau de bord.

🚗 Analogie : Node.js = moteur brut d'une voiture. Express.js = la voiture complète avec volant, freins, tableau de bord, clignotants. Vous pouvez conduire avec juste le moteur, mais c'est beaucoup plus difficile !

Ce qu'Express permet :

  • Créer des routes HTTP (GET, POST, PUT, DELETE)
  • Gérer les requêtes et réponses facilement
  • Ajouter des middlewares (fonctions intermédiaires)
  • Servir des fichiers statiques (HTML, CSS, images)
  • Construire des API REST complètes

⚖️ Sans Express vs Avec Express

❌ Sans Express (Node.js pur) — verbeux et difficile à maintenir :

const http = require('http'); const server = http.createServer((req, res) => { if (req.url === '/' && req.method === 'GET') { res.writeHead(200, {'Content-Type': 'text/plain'}); res.end('Bonjour !'); } else if (req.url === '/about') { res.writeHead(200, {'Content-Type': 'text/plain'}); res.end('À propos'); } else { res.writeHead(404); res.end('Non trouvé'); } }); server.listen(3000);

✅ Avec Express — simple, lisible, maintenable :

const express = require('express'); const app = express(); app.get('/', (req, res) => res.send('Bonjour !')); app.get('/about', (req, res) => res.send('À propos')); app.listen(3000);

🌟 Pourquoi utiliser Express ?

  • Simplicité : API claire et intuitive, courbe d'apprentissage douce
  • Rapidité : léger, performant, pas de surcharge inutile
  • Écosystème : des milliers de middlewares disponibles (npm)
  • Flexibilité : pas de structure imposée, vous organisez comme vous voulez
  • Communauté : le framework Node.js le plus populaire depuis 2010
  • REST API : parfait pour construire des backends pour React, Vue, mobile...

💡 Points clés à retenir

  • Express est un framework, pas un langage — il s'appuie sur Node.js
  • Il gère le routage HTTP de façon élégante
  • Les middlewares sont le cœur d'Express (on y reviendra)
  • Express est idéal pour créer des API REST
  • Il ne remplace pas Node.js, il le complète

✅ Prérequis

Avant de commencer, vérifiez que Node.js est installé sur votre machine :

# Vérifier la version de Node.js node --version # Exemple de sortie : v20.11.0 # Vérifier npm (Node Package Manager) npm --version # Exemple de sortie : 10.2.4

Si ces commandes ne fonctionnent pas, téléchargez Node.js sur nodejs.org (version LTS recommandée).

🚀 Initialiser et installer

# 1. Créer un dossier pour votre projet mkdir mon-projet cd mon-projet # 2. Initialiser le projet (crée package.json) npm init -y # 3. Installer Express npm install express # 4. Installer nodemon (rechargement automatique) npm install --save-dev nodemon
💡 nodemon surveille vos fichiers et redémarre automatiquement le serveur à chaque modification. Indispensable en développement !

📄 Le fichier package.json expliqué

{ "name": "mon-projet", ← nom du projet "version": "1.0.0", ← version actuelle "main": "app.js", ← fichier point d'entrée "scripts": { "start": "node app.js", ← npm start "dev": "nodemon app.js" ← npm run dev (avec rechargement) }, "dependencies": { "express": "^4.18.2" ← Express installé }, "devDependencies": { "nodemon": "^3.0.1" ← nodemon (dev seulement) } }

🗂️ Structure minimale du projet

mon-projet/ ├── node_modules/ ← dépendances (NE PAS TOUCHER) ├── app.js ← votre code principal ├── package.json ← configuration du projet ├── package-lock.json ← versions exactes (auto-généré) └── .gitignore ← fichiers à exclure de Git

Contenu du .gitignore :

node_modules/ .env
⚠️ Important : Ne jamais committer node_modules sur Git ! Ce dossier peut peser des centaines de Mo. Avec package.json, n'importe qui peut recréer node_modules avec npm install.

▶️ Lancer le projet

# Mode production (sans rechargement auto) node app.js # Mode développement (avec nodemon) npm run dev # Sortie attendue : # Serveur démarré sur http://localhost:3000

💡 Points clés à retenir

  • npm init -y crée package.json avec les valeurs par défaut
  • npm install express ajoute Express dans dependencies
  • --save-dev ajoute dans devDependencies (outils de dev uniquement)
  • nodemon redémarre le serveur automatiquement à chaque sauvegarde
  • Toujours ajouter node_modules/ dans .gitignore
  • Le script "dev" dans package.json permet de lancer nodemon avec npm run dev

🛠️ Création d'une application Express.js
Créez votre premier serveur Express, comprenez chaque ligne de code et ajoutez vos premières routes.

📝 Le fichier app.js minimal

// 1. Importer Express const express = require('express'); // 2. Créer l'application const app = express(); // 3. Définir le port const PORT = 3000; // 4. Définir une route GET sur '/' app.get('/', (req, res) => { res.send('🚀 Mon premier serveur Express !'); }); // 5. Démarrer le serveur app.listen(PORT, () => { console.log(`Serveur démarré sur http://localhost:${PORT}`); });

🔍 Explication ligne par ligne

  • require('express') → charge le module Express installé dans node_modules
  • express() → crée une instance de l'application avec toutes ses méthodes
  • req → objet requête : tout ce que le client envoie (URL, headers, body...)
  • res → objet réponse : ce qu'on renvoie au client (texte, JSON, status...)
  • app.get('/', ...) → écoute les requêtes GET sur l'URL '/'
  • res.send() → envoie une réponse texte/HTML au client
  • app.listen(PORT, callback) → démarre le serveur, callback exécuté quand prêt
🌐 Tester : Lancez le serveur avec npm run dev, puis ouvrez http://localhost:3000 dans votre navigateur. Vous devriez voir "🚀 Mon premier serveur Express !"

🗺️ Ajouter plusieurs routes

const express = require('express'); const app = express(); // Route principale app.get('/', (req, res) => { res.send('🏠 Page d\'accueil'); }); // Route /about app.get('/about', (req, res) => { res.send('ℹ️ Page À propos'); }); // Route /contact app.get('/contact', (req, res) => { res.send('📧 Page Contact'); }); // Route API qui retourne du JSON app.get('/api/status', (req, res) => { res.json({ status: 'OK', message: 'API opérationnelle' }); }); // Route 404 — TOUJOURS EN DERNIER app.use((req, res) => { res.status(404).send('❌ Page non trouvée'); }); app.listen(3000, () => console.log('Serveur sur port 3000'));
⚠️ Important : La route 404 avec app.use() doit toujours être déclarée en dernier. Express parcourt les routes dans l'ordre et s'arrête à la première qui correspond.

💡 Points clés à retenir

  • req = ce que le client envoie, res = ce qu'on lui renvoie
  • res.send() envoie du texte/HTML, res.json() envoie du JSON
  • Les routes sont lues dans l'ordre de déclaration
  • La route 404 (catch-all) doit être la dernière
  • app.listen() bloque et attend les connexions
  • Testez vos routes avec le navigateur ou un outil comme Postman

🗺️ Routage avec express.Router
Organisez vos routes dans des fichiers séparés grâce à express.Router pour un code propre et maintenable.

😰 Le problème : tout dans app.js

Quand votre application grandit, mettre toutes les routes dans app.js devient vite illisible :

// app.js — MAUVAISE PRATIQUE (tout mélangé) app.get('/api/projets', ...); app.post('/api/projets', ...); app.get('/api/projets/:id', ...); app.put('/api/projets/:id', ...); app.delete('/api/projets/:id', ...); app.get('/api/membres', ...); app.post('/api/membres', ...); // ... 50 routes plus tard, c'est illisible !
📁 Solution : express.Router() permet de créer des "mini-applications" pour chaque ressource, comme des dossiers pour organiser vos fichiers.

✅ Créer un router — routes/projets.js

// routes/projets.js const express = require('express'); const router = express.Router(); // GET /api/projets — liste tous les projets router.get('/', (req, res) => { res.json({ message: 'Liste de tous les projets' }); }); // POST /api/projets — créer un projet router.post('/', (req, res) => { res.json({ message: 'Projet créé' }); }); // GET /api/projets/:id — un projet spécifique router.get('/:id', (req, res) => { res.json({ message: `Projet ${req.params.id}` }); }); // Exporter le router module.exports = router;

🔌 Brancher le router dans app.js

// app.js const express = require('express'); const app = express(); // Importer les routers const projetsRouter = require('./routes/projets'); const membresRouter = require('./routes/membres'); // Brancher les routers avec leur préfixe app.use('/api/projets', projetsRouter); app.use('/api/membres', membresRouter); app.listen(3000);

Résultat des URLs disponibles :

GET /api/projets → liste des projets POST /api/projets → créer un projet GET /api/projets/:id → un projet GET /api/membres → liste des membres

⚠️ L'ordre des routes est crucial ! (Exemple du V6)

// routes/projects.js — ORDRE CORRECT const router = express.Router(); // ✅ Route spécifique AVANT la route dynamique router.get('/stats/summary', getProjectStats); // ✅ Route dynamique APRÈS router.get('/:id', getProjectById);
⚠️ Piège classique : Si /:id est déclaré avant /stats/summary, Express interprétera "stats" comme un id ! Toujours mettre les routes spécifiques avant les routes dynamiques.

🗂️ Structure recommandée du projet

app/ ├── routes/ │ ├── projets.js ← routes des projets │ ├── membres.js ← routes des membres │ └── contact.js ← routes du contact ├── controllers/ ← logique métier ├── models/ ← schémas Mongoose └── app.js ← point d'entrée

💡 Points clés à retenir

  • express.Router() crée un mini-routeur indépendant
  • On exporte le router avec module.exports
  • On branche le router dans app.js avec app.use('/préfixe', router)
  • Les routes spécifiques doivent être déclarées AVANT les routes dynamiques
  • Séparer les routes par ressource améliore la lisibilité et la maintenance

Leçon 5 / 11
🔑 Paramètres de route
Maîtrisez req.params pour les paramètres dynamiques et req.query pour les filtres optionnels.

🎯 Paramètres dynamiques avec :param

Les paramètres de route permettent de créer des URLs dynamiques. Le : indique un paramètre variable.

// Route avec paramètre :id router.get('/:id', (req, res) => { const id = req.params.id; res.json({ message: `Projet demandé : ${id}` }); }); // Exemples d'appels : // GET /api/projets/42 → req.params.id = "42" // GET /api/projets/abc123 → req.params.id = "abc123" // GET /api/projets/mon-projet → req.params.id = "mon-projet"
💡 Note : req.params.id est toujours une chaîne de caractères (string), même si c'est un nombre. Pensez à convertir avec parseInt() si nécessaire.

🔢 Plusieurs paramètres

// Route avec plusieurs paramètres router.get('/:categorie/:id', (req, res) => { const { categorie, id } = req.params; res.json({ categorie, id }); }); // GET /api/projets/web/42 // → req.params = { categorie: "web", id: "42" }

❓ Query strings — paramètres optionnels

Les query strings sont les paramètres après le ? dans l'URL. Ils sont optionnels et accessibles via req.query.

// GET /api/projets?type=Web&status=Terminé&search=portfolio router.get('/', (req, res) => { const { type, status, search } = req.query; res.json({ type, status, search }); }); // Résultat : // { type: "Web", status: "Terminé", search: "portfolio" } // Si un paramètre est absent, il vaut undefined // GET /api/projets?type=Web // → { type: "Web", status: undefined, search: undefined }

🔍 Exemple du V6 — getAllProjects avec filtres

// GET /api/projects?type=Web&status=Terminé&search=portfolio exports.getAllProjects = async (req, res) => { try { const { type, status, search } = req.query; const filter = {}; // Ajouter les filtres seulement s'ils sont définis if (type && type !== 'Tous') filter.type = type; if (status && status !== 'Tous') filter.status = status; if (search) filter.title = { $regex: search, $options: 'i' }; const projects = await Project.find(filter).sort({ createdAt: -1 }); res.json({ success: true, count: projects.length, data: projects }); } catch (err) { res.status(500).json({ success: false, message: err.message }); } };

💡 Points clés à retenir

  • req.params contient les paramètres de route (partie de l'URL)
  • req.query contient les paramètres après le ? (optionnels)
  • Les valeurs de req.params et req.query sont toujours des strings
  • Les routes spécifiques (/stats) doivent être avant les routes dynamiques (/:id)
  • req.query est parfait pour les filtres, tris et pagination

🔧 Middlewares
Comprenez le concept fondamental des middlewares Express : ces fonctions qui s'exécutent entre la requête et la réponse.

🤔 Qu'est-ce qu'un middleware ?

Un middleware est une fonction qui s'exécute entre la réception de la requête et l'envoi de la réponse. Il peut lire/modifier req et res, puis passer la main au suivant.

// Signature d'un middleware const monMiddleware = (req, res, next) => { // Faire quelque chose avec req ou res console.log('Middleware exécuté !'); next(); // ← OBLIGATOIRE pour passer au suivant };
✈️ Analogie : Imaginez la douane à l'aéroport. Chaque poste (contrôle passeport, scanner bagages, contrôle sécurité) vérifie quelque chose avant de vous laisser passer. Si un poste bloque, vous n'allez pas plus loin. C'est exactement le rôle des middlewares !

📋 Types de middlewares

TypeSyntaxeExemple
Applicationapp.use(fn)Logger, CORS
Routerouter.use(fn)Auth sur /api
Erreur(err, req, res, next)Gestion d'erreurs
Intégrésexpress.json()Parser le body JSON
Tiersnpm install corscors, morgan, helmet

📝 Middleware logger personnalisé

const logger = (req, res, next) => { console.log(`[${new Date().toISOString()}] ${req.method} ${req.url}`); // Exemple de sortie : // [2024-01-15T10:30:00.000Z] GET /api/projets next(); // ← Sans next(), la requête reste bloquée ! }; // Appliquer à toutes les routes app.use(logger);

🌐 Middlewares essentiels du V6

// 1. CORS — autoriser les requêtes cross-origin app.use(cors({ origin: process.env.FRONTEND_URL || 'http://localhost:5173', credentials: true })); // 2. express.json() — parser le body JSON // SANS ça : req.body = undefined app.use(express.json({ limit: '10mb' })); // 3. express.urlencoded() — parser les formulaires HTML app.use(express.urlencoded({ extended: true })); // 4. Middleware d'erreur — TOUJOURS EN DERNIER (4 paramètres) app.use((err, req, res, next) => { console.error(err.stack); res.status(500).json({ message: 'Erreur serveur interne' }); });
⚠️ Règle d'or : L'ordre des middlewares est crucial ! express.json() doit être déclaré AVANT les routes qui utilisent req.body. Le middleware d'erreur doit être le DERNIER.

🔄 Ordre d'exécution des middlewares

// Requête → Middleware 1 → Middleware 2 → Route → Réponse app.use(logger); // 1er exécuté app.use(express.json()); // 2ème exécuté app.use(cors()); // 3ème exécuté app.get('/', (req, res) => { // 4ème : la route res.send('OK'); }); app.use(errorHandler); // Dernier : gestion d'erreurs

💡 Points clés à retenir

  • Un middleware a toujours 3 paramètres : (req, res, next)
  • next() est obligatoire pour passer au middleware suivant
  • Sans next(), la requête reste bloquée indéfiniment
  • express.json() est indispensable pour lire req.body
  • Le middleware d'erreur a 4 paramètres : (err, req, res, next)
  • L'ordre de déclaration des middlewares est crucial

📦 req.body — données POST/PUT

// POST /api/projets // Body JSON envoyé : { "title": "Mon projet", "type": "Web" } router.post('/', (req, res) => { // Destructurer les données du body const { title, type, description } = req.body; // Validation manuelle simple if (!title) { return res.status(400).json({ error: 'Le titre est requis' }); } if (!type) { return res.status(400).json({ error: 'Le type est requis' }); } res.status(201).json({ message: 'Projet créé avec succès', data: { title, type, description } }); });
⚠️ Rappel : req.body nécessite le middleware express.json() déclaré avant les routes. Sans lui, req.body = undefined !

🔑 req.headers — en-têtes HTTP

// Lire le token d'authentification router.get('/profil', (req, res) => { const token = req.headers.authorization; // Exemple : "Bearer eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9..." const contentType = req.headers['content-type']; // Exemple : "application/json" res.json({ token, contentType }); });

🔍 Exemple complet — createProject du V6

// POST /api/projects exports.createProject = async (req, res) => { try { // req.body contient toutes les données envoyées const project = new Project(req.body); const saved = await project.save(); res.status(201).json({ success: true, data: saved }); } catch (err) { // Mongoose valide automatiquement les champs required res.status(400).json({ success: false, message: err.message }); } };

✅ Validation manuelle avant Mongoose

router.post('/', (req, res) => { const { title, type, description } = req.body; // Vérifier les champs obligatoires if (!title || !type) { return res.status(400).json({ error: 'Champs manquants', required: ['title', 'type'] }); } // Vérifier la longueur if (title.length > 100) { return res.status(400).json({ error: 'Titre trop long (max 100 caractères)' }); } // Tout est OK, continuer... res.status(201).json({ success: true }); });

💡 Points clés à retenir

  • req.params → paramètres de route (/api/projets/:id)
  • req.query → query string (/api/projets?type=Web)
  • req.body → corps JSON (POST/PUT) — nécessite express.json()
  • req.headers → en-têtes HTTP (Authorization, Content-Type...)
  • Toujours valider les données reçues avant de les utiliser
  • Retourner 400 Bad Request si les données sont invalides

🗄️ Qu'est-ce que MongoDB ?

MongoDB est une base de données NoSQL orientée documents. Au lieu de tables et de lignes (SQL), elle stocke des documents JSON dans des collections.

🔧 Qu'est-ce que Mongoose ?

Mongoose est un ODM (Object Document Mapper) pour Node.js. Il ajoute une couche de structure et de validation au-dessus de MongoDB.

👨‍💼 Analogie : MongoDB = entrepôt sans règles (on peut stocker n'importe quoi). Mongoose = chef d'entrepôt qui impose des règles : "les projets doivent avoir un titre, un type parmi ces valeurs, et une date de création".

Ce que Mongoose apporte :

  • Définir des schémas (structure des documents)
  • Validation automatique des données
  • Méthodes pratiques : find(), save(), findByIdAndUpdate()...
  • Middleware Mongoose (pre/post hooks)
  • Gestion des relations entre documents (populate)

// config/connectdb.js const mongoose = require('mongoose'); const connectDB = async () => { try { const conn = await mongoose.connect(process.env.MONGODB_URI); console.log(`✅ MongoDB connecté : ${conn.connection.host}`); } catch (error) { console.error(`❌ Erreur connexion : ${error.message}`); process.exit(1); // Arrêter le serveur si pas de DB } }; module.exports = connectDB;
